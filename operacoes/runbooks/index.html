Runbooks

Previous Next

Procedimentos passo a passo para operações e incidentes específicos.

O que é um Runbook?

Runbooks são procedimentos documentados que descrevem como realizar tarefas operacionais específicas ou responder a incidentes.

Como Usar Este Guia

  1. Identifique o runbook apropriado para sua situação
  2. Siga os passos na ordem descrita
  3. Documente qualquer desvio ou problema encontrado
  4. Atualize o runbook se necessário após conclusão

Runbooks Disponíveis

Deployment

Deploy para Produção

Quando usar: Deploy de nova versão para produção

Pré-requisitos: - [ ] Testes passando em staging - [ ] Aprovação do tech lead - [ ] Backup recente disponível - [ ] Janela de manutenção agendada (se aplicável)

Procedimento:

  1. Verificar Status Atual 

    # Verificar versão atual
kubectl get deployment application -n production -o jsonpath='{.spec.template.spec.containers[0].image}'

# Verificar health
curl https://api.yourproject.com/health

  2. Criar Backup 

    # Backup do banco de dados
./scripts/backup-database.sh --env=production

# Verificar backup
ls -lh backups/

  3. Deploy 

    # Fazer pull da nova imagem
docker pull registry.yourproject.com/application:v1.2.3

# Aplicar deployment
kubectl set image deployment/application \
  application=registry.yourproject.com/application:v1.2.3 \
  -n production

# Ou usando kubectl apply
kubectl apply -f k8s/production/deployment.yaml

  4. Monitorar Deploy 

    # Watch rollout status
kubectl rollout status deployment/application -n production

# Verificar pods
kubectl get pods -n production -w

  5. Smoke Tests 

    # Health check
curl https://api.yourproject.com/health

# Test endpoints críticos
./scripts/smoke-test.sh --env=production

  6. Verificar Métricas

  7. Abrir Grafana Dashboard
  8. Verificar error rate (deve estar < 0.1%)
  9. Verificar latency (P95 < 500ms)

  10. Verificar throughput

  11. Rollback (se necessário) 

    # Rollback automático
kubectl rollout undo deployment/application -n production

# Ou para versão específica
kubectl rollout undo deployment/application -n production --to-revision=2

  12. Documentar

  13. Atualizar changelog
  14. Notificar time no Slack
  15. Atualizar status page se houver downtime

Rollback Criteria: - Error rate > 1% por mais de 5 minutos - Latency P95 > 2s - Health check falhando - Bugs críticos reportados

Rollback de Deploy

Quando usar: Quando deploy causa problemas em produção

Urgência: Alta

Procedimento:

  1. Identificar Versão Anterior 

    # Listar revisões
kubectl rollout history deployment/application -n production

  2. Executar Rollback 

    # Rollback para revisão anterior
kubectl rollout undo deployment/application -n production

# Ou específica
kubectl rollout undo deployment/application -n production --to-revision=3

  3. Verificar 

    kubectl rollout status deployment/application -n production
curl https://api.yourproject.com/health

  4. Notificar

  5. Time de desenvolvimento
  6. Stakeholders
  7. Criar post-mortem

Database

Backup Manual do Banco

Quando usar: Antes de operações críticas ou mudanças de schema

Frequência: Sob demanda

Procedimento:

  1. Verificar Espaço em Disco 

    df -h /backup

  2. Criar Backup 

    # PostgreSQL
pg_dump -h database-host -U postgres -d production_db \
  -F c -b -v -f backup_$(date +%Y%m%d_%H%M%S).dump

# MySQL
mysqldump -h database-host -u root -p production_db \
  --single-transaction --quick --lock-tables=false \
  > backup_$(date +%Y%m%d_%H%M%S).sql

  3. Comprimir 

    gzip backup_*.dump

  4. Upload para Storage 

    aws s3 cp backup_*.dump.gz s3://yourproject-backups/database/

# Ou Google Cloud
gsutil cp backup_*.dump.gz gs://yourproject-backups/database/

  5. Verificar Integridade 

    # PostgreSQL
pg_restore --list backup_*.dump.gz

# MySQL
gunzip -t backup_*.sql.gz

  6. Documentar

  7. Nome do arquivo
  8. Timestamp
  9. Tamanho
  10. Localização

Restore do Banco

Quando usar: Recuperação de desastre ou correção de erro

Urgência: Crítica

Procedimento:

  1. STOP - Verificações Críticas
  2. Você tem certeza que precisa fazer restore?
  3. Você tem aprovação?
  4. Você tem backup válido?

  5. Aplicação está em modo manutenção?

  6. Colocar em Modo Manutenção 

    kubectl scale deployment/application --replicas=0 -n production

  7. Download do Backup 

    aws s3 cp s3://yourproject-backups/database/backup_20251024.dump.gz .
gunzip backup_20251024.dump.gz

  8. Restore 

    # PostgreSQL
pg_restore -h database-host -U postgres -d production_db \
  -v -c backup_20251024.dump

# MySQL
mysql -h database-host -u root -p production_db < backup_20251024.sql

  9. Verificar Dados 

    psql -h database-host -U postgres -d production_db

# Executar queries de verificação
SELECT COUNT(*) FROM users;
SELECT MAX(created_at) FROM users;

  10. Reativar Aplicação 

    kubectl scale deployment/application --replicas=3 -n production

  11. Smoke Tests 

    ./scripts/smoke-test.sh --env=production

Incidentes

Sistema Fora do Ar (P0)

Quando usar: Aplicação completamente indisponível

SLA: 15 minutos para resposta

Procedimento:

  1. Declarar Incidente (0-2min) 

    # No Slack
/incident Sistema fora do ar

  2. Triage Rápido (2-5min) 

    # Verificar pods
kubectl get pods -n production

# Verificar eventos
kubectl get events -n production --sort-by='.lastTimestamp' | tail -20

# Verificar logs
kubectl logs deployment/application -n production --tail=50

  3. Ação Imediata (5-10min)

Se pods não estão rodando: 

kubectl get pods -n production
kubectl describe pod <pod-name> -n production
kubectl rollout restart deployment/application -n production

Se database está inacessível: 

# Verificar conexões
kubectl exec -it application-pod -- nc -zv database-host 5432

# Restart do connection pool
kubectl rollout restart deployment/application -n production

Se load balancer tem problemas: 

# Verificar health checks
curl -I https://api.yourproject.com/health

# Verificar ingress
kubectl get ingress -n production

  1. Comunicação (Contínua)
  2. Atualizar status page
  3. Notificar stakeholders

  4. Updates a cada 15 minutos

  5. Resolução

  6. Aplicar fix
  7. Verificar métricas

  8. Confirmar resolução

  9. Post-Incident (24h depois)

  10. Criar post-mortem
  11. Identificar ações de melhoria

Alta Taxa de Erros (P1)

Quando usar: Error rate > 5%

SLA: 1 hora para resposta

Procedimento:

  1. Identificar Scope 

    # Verificar error rate por endpoint
# Grafana: Application Dashboard > Error Rate by Endpoint

# Verificar logs
kubectl logs deployment/application -n production | grep ERROR | tail -100

  2. Isolar Causa

Erros de validação: - Verificar mudanças recentes em schemas - Verificar dados de entrada

Erros de dependência: 

# Testar APIs externas
curl -I https://external-api.com/health

Erros de database: 

-- Verificar queries falhando
SELECT * FROM pg_stat_statements 
WHERE calls > 0 AND mean_exec_time > 1000
ORDER BY calls DESC;

  1. Mitigar

Se API externa está down: - Enable circuit breaker - Use cached data - Return degraded service

Se database query está falhando: - Add missing index - Optimize query - Add timeout

  1. Resolver
  2. Deploy fix
  3. Verify error rate drops
  4. Monitor for 30 minutes

Manutenção

Upgrade de Dependência Crítica

Quando usar: Atualização de linguagem, framework ou biblioteca crítica

Planejamento: 1-2 semanas

Procedimento:

  1. Preparação (1 semana antes)
  2. Ler changelog da nova versão
  3. Identificar breaking changes
  4. Criar branch para upgrade

  5. Atualizar em ambiente de dev

  6. Testes (3 dias antes)

  7. Rodar todos os testes
  8. Fazer testes manuais
  9. Load testing

  10. Deploy em staging

  11. Comunicação (2 dias antes)

  12. Notificar time
  13. Agendar janela de manutenção

  14. Preparar rollback plan

  15. Execução (Dia do upgrade) 

    # Backup
./scripts/backup-all.sh

# Deploy
kubectl apply -f k8s/production/

# Monitor
watch kubectl get pods -n production

  16. Validação

  17. Smoke tests pass
  18. Metrics normal
  19. No errors in logs

  20. Performance acceptable

  21. Rollback Plan 

    # Se algo der errado
kubectl rollout undo deployment/application -n production
# Restore dependencies antigas

Limpeza de Dados Antigos

Quando usar: Manutenção regular de dados

Frequência: Mensal

Procedimento:

  1. Identificar Dados 

    -- Logs mais antigos que 90 dias
SELECT COUNT(*) FROM logs 
WHERE created_at < NOW() - INTERVAL '90 days';

-- Sessões expiradas
SELECT COUNT(*) FROM sessions 
WHERE expires_at < NOW();

  2. Backup Antes de Deletar 

    ./scripts/backup-database.sh

  3. Delete em Batches 

    -- Delete em batches de 1000
DO $$
BEGIN
  LOOP
    DELETE FROM logs 
    WHERE id IN (
      SELECT id FROM logs 
      WHERE created_at < NOW() - INTERVAL '90 days'
      LIMIT 1000
    );

    EXIT WHEN NOT FOUND;

    -- Pausa entre batches
    PERFORM pg_sleep(1);
  END LOOP;
END $$;

  4. VACUUM 

    VACUUM ANALYZE logs;

  5. Verificar 

    -- Verificar espaço recuperado
SELECT pg_size_pretty(pg_total_relation_size('logs'));

Segurança

Resposta a Incidente de Segurança

Quando usar: Suspeita ou confirmação de brecha de segurança

Urgência: Crítica

Procedimento:

  1. IMEDIATO (0-15min)
  2. Notificar security team
  3. Isolar sistema afetado (se possível)
  4. Preservar evidências (logs, snapshots)

  5. NÃO deletar nada ainda

  6. Avaliação (15-60min)

  7. Identificar escopo do incidente
  8. Determinar dados afetados
  9. Identificar vetor de ataque

  10. Avaliar impacto

  11. Contenção (1-4h)

  12. Bloquear acesso malicioso
  13. Revogar credenciais comprometidas
  14. Aplicar patches de emergência

  15. Isolar sistemas afetados

  16. Erradicação (4-24h)

  17. Remover backdoors
  18. Aplicar fixes permanentes

  19. Fortalecer controles de segurança

  20. Recuperação (24-72h)

  21. Restaurar serviços
  22. Monitorar comportamento anormal

  23. Verificar integridade dos dados

  24. Documentação (Ongoing)

  25. Criar incident report
  26. Notificar partes afetadas (se aplicável)
  27. Reportar às autoridades (se necessário)
  28. Atualizar procedimentos de segurança
